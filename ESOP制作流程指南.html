📋 制作流程概览
整个 ESOP（电子标准作业程序）制作分为 7 个阶段，从拿到客户需求到最终以网页形式交付给客户。 核心思路：内容结构化 → 模板化生成 → 自动化部署。
1

需求分析 — 确定做哪些模块

输入：客户需求
1.1与客户确认模块清单
与客户开会确定需要制作哪些车间/部门的操作手册。
以美诗儿项目为例，最终确定 5 大模块： 🏭 注塑车间 · 🏭 组装车间 · ⚙️ 设备维保 · 🔧 模具维保 · 📋 质量检验
剔除的模块：IIoT 设备看板、系统基础配置（客户确认不需要）
💡 后续可能会新增模块，要预留扩展性
↓
1.2收集参考材料
从客户处获取制作 ESOP 所需的原始材料：
📄 Word 文档 — 鼎捷标准功能操作说明书（.doc 格式）
📄 PDF 简报 — 标准功能操作简报（如 PL102 设备稼动、PL205 模治具寿命管理）
🖼️ 截图素材 — 系统操作界面截图
💡 材料越完整，制作越高效
2

内容采集 — 从文档中提取操作步骤

输入：Word / PDF
2.1Word 文档解析
使用 python-docx 库读取 Word 文档，提取章节标题和操作步骤描述。 遇到加密文档可用 olefile + antiword 辅助读取。
python-docxantiword
↓
2.2PDF 参考文档解读
使用 PyMuPDF (fitz) 提取 PDF 文字和图片。
关键：逐页分析文字描述，找到与操作步骤匹配的内容，特别注意闭环流程（如设备稼动闭环、维修闭环、保养闭环）
PyMuPDFpdfminer.six
3

图片处理 — 提取并匹配截图

难度：⭐⭐⭐
⚠️ 最重要的步骤
图片必须与文字描述对应，这是客户验收时最容易发现问题的地方。方法：找到 PDF 中某操作步骤的文字页 → 提取该页的截图 → 验证截图内容与文字一致。
3.1从 PDF 提取截图
使用 page.get_images() 提取 PDF 中嵌入的图片，筛选出有效截图（排除装饰性元素）。 美诗儿项目中从 2 个 PDF 提取了 140+ 张截图。
↓
3.2图文匹配验证
匹配方法：
① 找到 PDF 中描述某操作的页面 → ② 提取该页主要截图 → ③ 验证截图是否与文字描述一致
示例：PDF P59 写"在作业人员工作平台找到设备" → 对应 ESOP 步骤 1.1 "进入工作平台"
💡 客户反馈"图片不对"是最常见的修改项
↓
3.3图片整理命名
统一命名规则：fig-01.png、fig-02.png 顺序编号。
放入对应模块的 images/ 子目录。
images/equipment/fig-01~16.png
4

代码开发 — 编写 HTML 生成器

核心：generate.py
4.1设计数据结构和模板
用 Python dict 定义模块内容（CONTENT 变量），包含：
title · dept · icon · img_dir · scenario · steps · notes
HTML 模板使用 str.replace() 渲染
generate.py (~580 行)
↓
4.2自动生成 SVG 流程图
根据步骤数据自动绘制 横向流程图：主步骤→子步骤，箭头连线自动计算坐标。
支持主步骤之间的水平箭头和主步骤到子步骤的垂直箭头。
↓
4.3CSS 样式 + JS 交互
🎨 固定侧边栏导航 · 全宽响应式布局 · 图片点击放大 · 滚动章节高亮
🎨 深色主题侧边栏 · 卡片式布局 · 磨砂玻璃效果
css/style.cssjs/main.js
5

内容编写 — 结构化录入

最耗时阶段
5.1编写步骤描述
每条步骤包含：id·name·desc(作业说明)·remark(备注)·images(图片列表)
描述要用户看得懂，避免技术术语，用操作人员能理解的语言。
💡 多和客户确认描述是否准确
↓
5.2重点：闭环流程设计
设备维保的例子：
🔄 维修闭环：加工中→故障→维修→加工中
🔄 保养闭环：加工中→保养→加工中
模具维保的例子：
🔄 异常发现→叫修→签收→维修作业→完工确认→恢复可用
💡 闭环思维是客户最看重的
↓
5.3运行一行命令，生成全部页面
python generate.py → 自动生成所有模块的 HTML 文件到 pages/ 目录
同时生成首页 index.html（模块卡片导航）
✅ 一次运行生成 6 个页面
6

校验与修正 — 迭代优化

循环多次
6.1本地预览校验
✅ 打开 HTML 页面检查排版
✅ 验证图片加载、图文对应
✅ 检查流程图箭头方向
✅ 验证侧边栏导航
↓
6.2客户反馈修正
🔄 客户查看 → 提出修改意见 → 调整内容 → 重新生成 → 再次验证
常见的修正：图片不对、步骤描述不准确、缺少某些环节、流程需要调整
💡 预留 2-3 轮迭代时间
7

部署上线 — 让客户访问

多种方式
📌 选择方案
根据客户需求选择部署方式。美诗儿项目最终方案：Git 版本管理 + GitHub + Cloudflare Pages CDN 加速。

📁 方案A：本地服务器

适合开发调试、临时展示，电脑不能关机

python -m http.server 8000

🐙 方案B：GitHub Pages

适合国内客户，推送即部署

git push origin main

☁️ 方案C：Cloudflare Pages

全球 CDN 加速，自动 HTTPS，推荐正式发布

连接 Git → 自动部署

📦 方案D：压缩包发送

最直接，客户解压双击 index.html 即可

压缩 → 微信/邮件发送
7.1Git 版本管理
git init → git add -A → git commit -m "描述" → git push
SSH 密钥认证：ssh-keygen → GitHub Settings 添加公钥 → 免密推送
↓
7.2Cloudflare Pages 部署
两种方式：
① Cloudflare 控制台 → Pages → 连接 Git 仓库 → 自动部署（不需要令牌）
② 命令行：npx wrangler pages deploy . --project-name=xxx（需要 API 令牌）
✅ 部署后自动获得 xxxx.pages.dev 网址
↓
7.3最终交付
🎯 客户通过网址直接访问，无需安装任何软件
🎯 手机/电脑/平板均可打开
🎯 后续更新：修改 content → 运行 generate.py → git push → 自动更新

💡 核心理念
📌 内容与样式分离
所有操作步骤写在 Python dict 中，样式写在 CSS 中，模板写在 HTML 模板中。 修改内容只需改 CONTENT 字典 → 运行 generate.py → 重新部署，3 步完成更新。
📌 自动化思维
不用手动写每个 HTML 页面，而是写一个生成器。新增模块只需在 CONTENT 中添加一个字典条目， 生成器自动处理：渲染页面 + 生成流程图 + 更新首页导航 + 生成侧边栏。
📌 版本控制
每次修改都用 Git 提交，CHANGELOG.md 记录每次变更内容。出现问题可以随时回退到上一个版本。 推送到 GitHub 后 Cloudflare 自动部署，客户立即看到最新版。
